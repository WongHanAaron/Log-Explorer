## Summary

This work extracts the two‑column browse‑and‑edit pattern implemented for
Log File Path configs into a reusable framework.  The generic infrastructure
provides a host‑side helper class plus a React-based webview component that can
be parameterised by a `ConfigCategory` and schema-specific rendering logic.
Panels built on the framework automatically display a searchable list of
config names, react to external adds/deletes, prompt when unsaved changes
would be lost, and post standard messages (`selectConfig`, `configData`,
`updateConfig`, etc.).  The goal is to enable future editors (alert rules,
export formats, etc.) to be created with minimal boilerplate, improving
consistency and reducing duplication.

`@vscode/test-electron`.
**Target Platform**: Desktop VSCode (Windows/macOS/Linux).
**Project Type**: VSCode extension (single project).
**Performance Goals**: Name list filtering sub‑200 ms with up to 1k entries;
config-change propagation within 2 seconds.
**Constraints**: Must operate within VSCode webview CSP; avoid blocking
extension host; support multiple panels concurrently.
**Scale/Scope**: Core framework ~1k‑2k LOC; reused by potentially multiple
future features.
