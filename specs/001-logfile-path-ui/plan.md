## Summary

The Log File Path configuration panel currently presents a single
editor for a single configuration at a time.  The requested enhancement splits
that view into two resizable columns: a searchable, liveвЂ‘updated list of all
stored fileвЂ‘path configs on the left (with filtering) and the existing editor on
the right.  Clicking a name loads it for editing; saves write back to
`ConfigStore`.  The list must refresh when configs are added or removed by
other services.  Additionally, the `ConfigStore` will log every change via
`OutputLogger` per the updated requirement.  This change improves discoverability
and makes bulk maintenance of path configs more productive.

## Technical Context

**Language/Version**: TypeScript 5.x targeting Node/ES2021 in a VSCode
extension.  Webview UI uses React for existing panels.
**Primary Dependencies**: vscode API (`@types/vscode`), React/tailwind libs used
by other webviews; `ConfigStore` service in `src/services`.  No new external
packages anticipated (reuse existing React setup).
**Storage**: Configurations are persisted via `ConfigStore` to the workspace
filesystem; panel reads via that service.  No new storage mechanism.
**Testing**: Unit tests with Mocha/Sinon/Chai (`test/unit`); integration tests
via `@vscode/test-electron` harness.  Add UIвЂ‘level tests by simulating
webview messages if needed.
**Target Platform**: Desktop VSCode (Windows, macOS, Linux).
**Project Type**: Desktop-app plugin (VSCode extension).
**Performance Goals**: List rendering should handle hundreds of entries with
subвЂ‘second filter response.  Update notifications must propagate within a few
seconds when store changes occur.
**Constraints**: Must work within the webview security model (CSP, message
passing), maintain existing form layout, and degrade gracefully on narrow
panels.  Avoid blocking the extension host during list refresh.
**Scale/Scope**: A single feature affecting only the Log File Path config
panel; approximately 1вЂ“2k LOC changes concentrated in `src/panels` and
`src/services`.

## Tasks

1. **Augment `ConfigStore` with logging**
   - Add `OutputLogger` import and emit a log entry on create/update/delete
     methods.  Include config name and action string.
   - Write unit tests verifying log entries are generated when the corresponding
     asynchronous operations succeed, using a fake logger.
2. **Enhance message protocol**
   - Update `contracts/panel-protocol.md` to define `configListChanged` and
     `init` messages (already done); ensure the spec reflects logging behavior
     where applicable.
3. **Update host panel (`LogFilePathPanel.ts`)**
   - Modify initialization to fetch list with
     `ConfigStore.listConfigNames(Filepath)` and send `init` message.
   - Register listener on `ConfigStore.onDidChange` and forward
     `configListChanged` to the webview whenever name list changes.
   - Handle incoming messages: `selectConfig`, `updateConfig`, `createConfig`,
     `deleteConfig` by calling appropriate store methods and replying with
     `configData` or confirmation messages.
   - Add keyboard navigation helpers (arrow keys, enter) for the React list
     component if not already present.
4. **Build React splitвЂ‘view UI**
   - Create `ConfigList` component featuring an input box and filtered `<ul>`
     based on props.  Use `useEffect` to update when `props.names` changes.
   - Compose `SplitView` in `main.tsx` with `ConfigList` on the left and the
     existing edit form (extracted from current panel) on the right.
   - Wire message sending (`selectConfig`, `updateConfig`, etc.) and handle
     responses (`configData`, `configListChanged`).
   - Implement search filtering (caseвЂ‘insensitive substring) with debounced
     input (200вЂЇms).  Add an empty-state message when list is empty.
5. **Keyboard accessibility & focus management**
   - Ensure search box has `tabindex=0` and captures keystrokes.
   - Implement arrow-key navigation to move selection in `ConfigList` and
     propagate Enter key to trigger selection.
   - Add appropriate `aria` attributes for screen readers (e.g. `aria-live`
     on list updates).
6. **External-change handling**
   - In host, when a config is deleted that matches the currently loaded one,
     send a message to clear the editor or load another available config.
   - In React form, display an error banner if the loaded config disappears.
7. **Responsive layout**
   - Use CSS flex and `max-width` rules to allow the two columns to share
     space and wrap or scroll at narrow widths.  Add a mobile friendly fallback
     if time permits (P3).
8. **Integration tests**
   - Write an integration test opening the panel, populating the store,
     selecting names, editing them, and verifying the store changes.
   - Simulate external creation/deletion via `ConfigStore` and assert that the
     webview list updates (using message spying or DOM queries).
9. **Documentation & quickstart update**
   - Add instructions to `quickstart.md` on using the split view and verifying
     logging via the output channel.
10. **Cleanup & review**
    - Remove any leftover TODO comments, ensure TypeScript compilation passes,
      and run `npm run lint` if available.
    - Create `specs/001-logfile-path-ui/tasks.md` checklist reflecting above
      items for Phase 2 tracking.
